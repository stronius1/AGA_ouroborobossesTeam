## Режим разработчика
Разработка фронта в режиме backend:

Чтобы не пересобирать фронт каждый раз, можно запустить отдельно бек и отдельно фронт. Для этого в package.json заведены 2 скрипта:
- dev-backend-ui-up - запускает ui который смотрит на бекенд на 3000 порту, в режиме разработки (serve)
- dev-backend-bh-up - запускает бекенд на 3000 порту

Теперь можно дорабатывать фронт не пересобирая его каждый раз

## Сборка и публикация

### Публикация в NPM
На данный момент поддерживается публикация в два `npm` реестра:
- `gitverse.ru` (api: https://gitverse.ru/api/packages/seafteam/npm/)
- `npmjs.org` (api: https://registry.npmjs.org/)

В `npm` реестр публикуется пакет `seaf-archtool-core-plugin-suite`, подключаемый в проекты [seaf-archtool-ideplugin-vscode](https://gitverse.ru/seafteam/seaf-archtool-ideplugin-vscode) и [seaf-archtool-ideplugin-jetbrains](https://gitverse.ru/seafteam/seaf-archtool-ideplugin-jetbrains). Он содержит артефакт, необходимый для работы плагинов, а также дополнительную мета-информацию.

Перед публикацией выполняется сборка
```bash
npm run plugin:build
```
Далее, команда для публикации (по умолчанию в `gitverse.ru`)
```bash
export NPM_TOKEN=*** # Токен реестра
npm run plugin:publish
```

Можно явно указать реестр из поддерживаемых
```bash
npm run plugin:publish -- --registry=<gitverse.ru|npmjs.org>
```

Также можно переопределить текущую версию для публикации
```bash
npm run plugin:publish -- --version=<a.b.c|a.b.c-suffix.x|...>
```

> **ЗАМЕЧАНИЕ**:
> В реестре `npmjs.org` каждая версия допускается для публикации ЕДИНОЖДЫ! Даже если вы удалите ее из реестра, повторно залить не получиться. В реестре `gitverse.ru` на данный момент (2026-07-01) повторные залития допустимы, однако лучше не пользоваться такой пратикой. Для удаления можно воспользоваться командой
```bash
npm unpublish @seafteam/seaf-archtool-core-plugin-suite@<version> --@seafteam:registry=<registry-api>
```
предварительно указав в `.npmrc` нужный токен.

> **ЗАМЕЧАНИЕ**:
> Тег `latest` всегда присваивается последней загруженной СТАБИЛЬНОЙ версии (без пререлизных суффиксов), не зависимо от того, загружались ли после пререлизные версии (`*-dev`,`*-rc`,...). Однако в реестре `npmjs.org` тег `latest` может автоматически назначиться для пререлизной версии, если только, ни одной релизной версии еще не существует в реестре.

### Docker
Основная часть описана уже в разделе [сборка](README.md#сборка) главного Readme. Здесь необходимо дополнить про сборку/публикацию через специальный инструмент `Docker Bake`.

#### Docker Bake (рекомендованный способ публикации)
Готовая конфигурация для `bake` позволяет сразу собирать и публиковать образы в используемые реестры. Для этого сначала логинимся в реестре `gitverse.ru`
```bash
docker login -u seafteam gitverse.ru
```
либо в реестре `docker.io`
```bash
docker login -u seaf
```
Далее используем `Docker Bake`:
```bash
TAGS='latest,версия.1,...' docker buildx bake -f .docker/bake.hcl <docker_io|gitverse_ru> --push
```
В `TAGS` через запятую перечисляются нужные таги для публикации. Если `TAGS` не указан, то образ публикуется под тагом `unknown`.

> **ЗАМЕЧАНИЕ**:
> В конфигурации для `Bake` указано использование кэша слоев, что бы с нуля не пересобирать все. Если требуется полная пересборка, то проще всего добавить в команду явное указание на это, например, так:
```bash
TAGS='latest,версия.1,...' docker buildx bake -f .docker/bake.hcl <docker_io|gitverse_ru> --set '*.no-cache=true' --push
```

#### Особенность публикации Docker-образов в GitVerse
В реестр `gitverse.ru` можно загрузить только образы, у которых, слои весят не больше 100мб. В связи с этим, в [Dockerfile](.docker/build/Dockerfile) добавлен аргумент `GITVERSE=true`, который позволяет дробить тяжелые слои. На итоговый вес образа это никак не влияет. Таким образом, для единообразия образов, `Docker Bake` для всех используемых реестров производит сборку с аргументом `GITVERSE=true`.

#### Docker build/buildx
Также можно собтирать и публиковать стандартным способом через `build/buildx`, добавляя в соответствующие команды флаг `--push`. Также, как указано [выше](#особенность-публикации-в-gitverse), необходимо добавлять в команду аргумент для сборки `--build-arg GITVERSE=true`.
