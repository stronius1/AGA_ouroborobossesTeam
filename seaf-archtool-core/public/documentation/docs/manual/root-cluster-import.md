# Главный (root) манифест для кластера

Главное отличие от манифеста для backend режима в том как выглядит root манифест

## Root манифест

Раньше в нем было просто перечисление импортов различных манифестов, теперь он должен содержать список объектов с
правами

Было так:

```yaml
imports:
  - ka_dzo_berezka/dochub.yaml
  - ka_dzo_dub/dochub.yaml
  - ka_dzo_osinka/dochub.yaml
```

Стало так:

```yaml
imports:
  - root: ka_dzo_berezka/root.yaml
    permission: ecogroup.berezka
    alias: berezka
    warmupNeeded: true
    title: ИП "Березка"
    repos:
      - type: kadzo
        url: bitbucket:repo_ecogroup:pj_berezka
  - root: ka_dzo_dub/root.yaml
    permission: ecogroup.dub
    alias: dub
    name: ООО "Дубок"
    repos:
      - type: kadzo
        url: bitbucket:repo_ecogroup:pj_dub
  - root: ka_dzo_osinka/root.yaml
    permission: ecogroup.osinka
    alias: Осинка
    repos:
      - type: kadzo
        url: bitbucket:repo_ecogroup:pj_osinka
```

Импорты теперь это объект и он может содержать несколько атрибутов:

| Название     | Обязательность | Дефолт | Описание                                                                                                                                                            |
|--------------|----------------|--------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| root         | Да             |        | Путь к файлу с импортом отдельной архитектуры                                                                                                                       |
| permission   | Да             |        | Права, которые дают доступ к архитектуре из import (сравниваются с jwt)                                                                                             |
| alias        | Да             |        | Название архитектуры, которое используется в запросах, например при перезагрузке                                                                                    |
| title        | Нет            |        | Название архитектуры, которое будет отображаться в меню при выборе, какую архитектуру отображать (если не указан то будет использован alias)                        |
| warmupNeeded | Нет            | false  | true/false признак необходимости рассчитать datasets в момент загрузки архитектуры, а не при запросе пользователя                                                   |
| repos        | Нет            |        | Массив данных для сохранения архитектур, указывается если необходимо выгружать разные типы архитектур инструментами                                                 |
| repos.type   | Да             |        | Тип архитектуры (kadzo, оценки)                                                                                                                                     |
| repos.url    | Да             |        | Ссылка на git репозиторий используемый для сохранения указанного типа архитектуры в том же формате как и VUE_APP_DOCHUB_ROOT_DOCUMENT но без указания файла и ветки |

Если в настройках окружения ролевая модель выключена, но запущен режим кластера, то в root манифесте
допустимо указать только 1 объект архитектуры (один импорт). Т.к. ролевая модель выключена у всех будет доступ к нему.

#### Массив repos:
В релизе >=3.29 используется для отправки архитектуры через плагин. Сервис отправки будет запрашивать данные архитектуры у core сервиса и на основе отправленного
типа архитектуры будет искать подходящий repos.type в массиве repos. Далее архитектура будет сохранена по указанному (repos.url) git url

Сам массив не обязательный, но при наличии элементов они обязательно должны содержать атрибуты type и url

#### Перезагрузка
Для перезагрузки так же используется запрос PUT seaf-core/api/core/storage/reload описание которого можно найти на странице со swagger api (SEAF.ArchTool/Руководство/API SEAF.ArchTool)

#### Отображение одного манифеста за раз
Если система запущена в режиме кластера и включена ролевая модель, тогда активируется режим просмотра одного импорта (архитектуры). 
Это значит, что на UI в меню появляется кнопка для выбора импорта в виде "Архитектура: {{ archName }}".
При нажатии на выбор импорта, пользователь будет видеть список доступных ему импортов (архитектур) и сможет переключаться между ними.
Выбранная архитектура привязана к вкладке (в разных вкладках можно открыть разные архитектуры). В Url проставляется параметр выбранной архитектуры, что позволяет ей делиться.


## Переменные окружения для управления ролевой моделью:

| Название                      | Дефолт | Описание                                                                           |
|-------------------------------|--------|------------------------------------------------------------------------------------|
| VUE_APP_DOCHUB_ROLES_MODEL_V2 | N      | Ролевая модель v2 основанная на правах в root манифесте Y - включено N - выключено |
