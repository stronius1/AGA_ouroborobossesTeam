# Пакеты. Руководство пользователя

## Что такое пакет

Пакет - самодостаточная, отчуждаемая часть кода архитектуры. Он может содержать все то же, что обычный код.
Целью его создания является переиспользование архитектурного кода в сторонних архитектурных кодовых базах
в качестве зависимости.

Например, можно создать пакет описывающий вашу корпоративную архитектурную нотацию и передать командам как 
общий стандарт описания архитектуры. Пример создания нотации C4 Model можно посмотреть 
[здесь](https://github.com/rpiontik/DocHubExamples/tree/main/src/C4Model).

Рекомендуется следующая файловая структура пакета:
```
.\C4Model\              - Идентификатор пакета
.\C4Model\Entities\     - Сущности, которые содержит в себе пакет
.\C4Model\Validators\   - Валидаторы
.\C4Model\Datasets\     - Источники данных
.\C4Model\Docs\         - Документы
.\C4Model\dochub.yaml   - Корневой манифест
.\C4Model\README.md     - Описание пакета
```

## Структура файла пакета

Пакет описывается в YAML-файле и начинается с раздела $package.

Пример:
``` yaml
$package:
  mypack:
    version: 1.1
    name: My Package
    description: Пример пакета
    license: MIT

imports:
  - mm/root.yaml

entities:
  myEntity:
    name: Example entity
```
После раздела $package в файле могут располагаться любые разделы манифеста.

## Описание пакета

Минимальное объявление пакета:
``` yaml
$package:
  mypack:
    version: 1.1
```
### Обязательные параметры

* packageId (mypack в примере выше) — идентификатор пакета.
* version — версия пакета.

### Рекомендуемые параметры

Хотя они не обязательны, рекомендуется указывать:

* name — название пакета;
* description — описание назначения пакета;
* license — лицензию использования.

Пример:
``` yaml
$package:
  mypack:
    version: 1.1
    name: User Management
    description: Пакет с сущностями пользователей и ролей
    license: Apache License, Version 2.0
```
Эти поля не влияют на работу пакета, но делают его понятнее для других пользователей.

## Подключение пакетов

Каждый пакет объявляется в отдельном файле.

Пакеты подключаются через imports в корневом манифесте или в других пакетах:

``` yaml
imports:
  - packages/user-management.yaml
  - packages/core.yaml
```

### Порядок обработки

1. Система валидирует объявление пакета.
2. Проверяет зависимости.
3. Если все зависимости разрешены, обрабатывает содержимое пакета.
4. Если зависимости не разрешены, обработка пакета прекращается.


## Правила использования пакетов

При работе с пакетами действуют следующие правила:

* идентификатор пакета должен быть уникальным;
* версия пакета должна быть указана;
* нельзя подключать один и тот же пакет в разных версиях;
* пакет может содержать архитектурный код (imports, entities или другие разделы).
